# مجلس مستشار أمد للوعي المالي
### AMD Financial Council — On-Premise Multi-Agent Advisory System

نظام وكلاء ذكاء اصطناعي متعدد (Multi-Agent) يحاكي مجلساً استشارياً مالياً
مكوّناً من ثلاثة مستشارين متخصصين، يعمل **بالكامل محلياً (On-Premise)**
دون أي اعتماد على واجهات برمجة تطبيقات سحابية — لضمان أعلى مستوى ممكن
من الخصوصية وحماية البيانات المالية الحساسة للمستخدم.

---

## 1. لمحة عن المشروع

يقدّم "مجلس مستشار أمد" تجربة استشارة مالية تفاعلية عبر ثلاث شخصيات
ذكاء اصطناعي متخصصة تتناقش فيما بينها ومع المستخدم:

| الوكيل | التخصص | الدور |
|---|---|---|
| 🟢 **سلمان** | المخطط المالي | الميزانيات، الادخار، ترتيب الأولويات المالية |
| 🔵 **نورة** | محللة المخاطر | تقييم الاستثمارات، الأسهم، العقار، إدارة المخاطر |
| 🟠 **فهد** | خبير السلوك المالي | التحيزات السلوكية، عادات الإنفاق، الالتزام الذاتي |

### لماذا معمارية محلية 100%؟

خلافاً للحلول التقليدية المعتمدة على مزوّدي استدلال سحابيين (مثل
OpenRouter أو OpenAI)، يعتمد هذا النظام **حصراً** على مكوّنات تُشغَّل
داخل البنية التحتية الخاصة بالجهة (On-Premise)، بحيث لا تغادر أي بيانات
مالية للمستخدم الشبكة المحلية أبداً — وهو ما يخدم مباشرة متطلبات حماية
البيانات لدى **سدايا (SDAIA)** و**ساما (SAMA)**.

### التقنيات الأساسية

- **[Ollama](https://ollama.com)** — محرك استدلال محلي بالكامل (نماذج مثل
  `llama3.1`)؛ يستبدل أي اتصال بواجهات برمجة تطبيقات سحابية للدردشة
  والتقييم الديناميكي وتوليد الأجوبة.
- **[LangGraph](https://langchain-ai.github.io/langgraph/)** — يدير سير
  عمل المجلس عبر **توجيه ديناميكي متسلسل (Sequential Dynamic Routing)**:
  كل وكيل عقدة (Node) مستقلة تُنفَّذ بالتتابع، وتقرر بنفسها دخولياً هل
  السؤال ضمن تخصصها أم تتجاوزه للوكيل التالي — بدل أن يردّ الجميع دفعة
  واحدة على كل سؤال.
- **[ChromaDB](https://www.trychroma.com/)** — قاعدة بيانات متجهة محلية
  تُشغّل نظام **RAG متعدد المسارات (Segregated RAG)**: قاعدة معرفة
  مستقلة لكل وكيل (`planner_docs` / `risk_docs` / `behavior_docs`) لمنع
  تسرّب سياق تخصص إلى آخر.
- **صندوق أدوات آمن (Secure Sandbox) + بوابة أمان (Gateway Proxy)** —
  أدوات معزولة (أسعار الأسهم، مؤشر العقار، الربط البنكي، تصنيف
  المصاريف) تمرّ استجاباتها الحساسة إلزامياً عبر طبقة Middleware تُنقّح
  (Anonymize) أي أرقام هوية أو حسابات قبل خروجها للوكيل أو المستخدم.

### مبدأ معماري مهم: لا رفع ملفات من المستخدم

المستخدم النهائي يتفاعل عبر **الدردشة النصية فقط**. تغذية قواعد
المعرفة (RAG) بملفات PDF مهمة إدارية يقوم بها المطوّر (Backend Admin)
حصراً عبر سكريبت مستقل (`ingest_data.py`)، بمعزل تام عن دورة حياة
الدردشة الحية.

## 2. هيكل الملفات

```
amd-council/
│
├── server.py                  # الخادم الرئيسي (FastAPI): يدير Ollama،
│                               #   يبني ويشغّل StateGraph من LangGraph
│                               #   للتوجيه التسلسلي، ويبث الردود عبر SSE
│
├── rag_engine.py               # محرك الاسترجاع (Retrieval-Only): يحوّل
│                               #   سؤال المستخدم لـ Embedding محلي عبر
│                               #   Ollama ويستعلم في مسار ChromaDB الخاص
│                               #   بكل وكيل
│
├── ingest_data.py               # سكريبت المطوّر (Offline/Admin): يقرأ
│                               #   ملفات PDF من data/، يقسّمها، يضمّنها،
│                               #   ويحفظها في ChromaDB — لا علاقة له
│                               #   بواجهة الدردشة
│
├── agents/
│   ├── __init__.py
│   └── prompts.py               # الشخصيات (System Prompts) الثلاث:
│                               #   قواعد الامتثال، تعليمات الاعتماد
│                               #   الحصري على سياق RAG، وتعليمات
│                               #   استدعاء الأدوات (Tool Calling)
│
├── sandbox/
│   ├── __init__.py
│   └── tools.py                 # بيئة الأدوات الآمنة: أسعار الأسهم
│                               #   (yfinance)، مؤشر العقار (Mock)،
│                               #   الربط البنكي (Mock + Gateway Proxy
│                               #   لتنقيح PII)، تصنيف المصاريف
│
├── public/                      # الواجهة الأمامية (دردشة نصية فقط)
│   ├── index.html               #   الصفحة الرئيسية — بدون أي واجهة رفع ملفات
│   ├── app.js                   #   منطق البث الحي (SSE) وآلية الطابور
│   └── style.css                #   التنسيق والألوان
│
├── data/                        # مجلدات مصدر ملفات PDF (يملؤها المطوّر يدوياً)
│   ├── planner_docs/             #   مستندات سلمان (تخطيط مالي)
│   ├── risk_docs/                #   مستندات نورة (مخاطر واستثمار)
│   └── behavior_docs/            #   مستندات فهد (سلوك مالي)
│
├── chroma_store/                # قاعدة بيانات ChromaDB (تُولَّد تلقائياً
│                               #   بعد أول تشغيل لـ ingest_data.py)
│
├── requirements.txt             # الاعتماديات (Ollama, LangGraph, LangChain,
│                               #   ChromaDB, yfinance... بدون أي مكتبات سحابية)
├── .env.example                 # نموذج متغيرات البيئة — انسخه إلى .env
├── .gitignore
└── README.md                    # هذا الملف
```

## 3. دليل التثبيت والتشغيل المتسلسل

> **المتطلبات الأساسية:** Python 3.10+، و[Ollama](https://ollama.com/download)
> مثبّت على نفس الجهاز أو على خادم يمكن الوصول إليه ضمن الشبكة المحلية.

### الخطوة 1 — تثبيت الاعتماديات

```bash
# (اختياري لكن مُستحسن) إنشاء بيئة افتراضية
python3 -m venv .venv
source .venv/bin/activate        # على Windows: .venv\Scripts\activate

# تثبيت كل الاعتماديات
pip install -r requirements.txt
```

### الخطوة 2 — تشغيل خادم Ollama وتنزيل النماذج المطلوبة

```bash
# شغّل خادم Ollama (في نافذة طرفية منفصلة، يبقى يعمل في الخلفية)
ollama serve

# في نافذة طرفية أخرى: نزّل نموذج المحادثة الرئيسي
ollama pull llama3.1

# نزّل نموذج التضمين المستخدم في RAG
ollama pull nomic-embed-text
```

تأكد أن الخادم يستجيب قبل المتابعة:

```bash
curl http://localhost:11434/api/tags
```

### الخطوة 3 — إعداد متغيرات البيئة (اختياري)

```bash
cp .env.example .env
# عدّل القيم داخل .env إذا لزم (مثل تغيير OLLAMA_MODEL أو المنفذ PORT)
```

### الخطوة 4 — تغذية قواعد المعرفة (RAG)

ضع ملفات PDF الخاصة بكل تخصص داخل المجلد المناسب:

```
data/planner_docs/   ← أدلة الميزانية، خطط الادخار...
data/risk_docs/      ← تقارير الاستثمار، تحليلات السوق...
data/behavior_docs/  ← مراجع علم النفس المالي السلوكي...
```

ثم شغّل سكريبت التغذية:

```bash
# يغذي المسارات الثلاثة دفعة واحدة
python ingest_data.py

# أو لتغذية مسار واحد فقط (مثلاً بعد إضافة ملف جديد لنورة)
python ingest_data.py --agent risk

# لمسح مجموعة معينة وإعادة تغذيتها من الصفر بدل الإضافة التراكمية
python ingest_data.py --agent planner --reset
```

> إذا لم تُغذَّ إحدى القواعد بعد، يعمل النظام بشكل طبيعي؛ الوكيل
> المعني يعتمد على معرفته العامة فقط مع التنبيه لذلك ضمنياً في رده
> (راجع `RAG_GROUNDING_RULES` في `agents/prompts.py`).

### الخطوة 5 — تشغيل الخادم الرئيسي

```bash
python server.py
# أو عبر uvicorn مباشرة مع إعادة التحميل التلقائي أثناء التطوير:
uvicorn server:app --host 0.0.0.0 --port 3000 --reload
```

افتح المتصفح على:

```
http://localhost:3000
```

### الخطوة 6 — التحقق من صحة الإعداد

```bash
curl http://localhost:3000/api/health
```

يعرض هذا المسار حالة الاتصال بـ Ollama، النموذج المفعّل، وحالة كل
قاعدة معرفة RAG (عدد المقاطع المخزّنة لكل وكيل) — مفيد للتأكد من نجاح
خطوة `ingest_data.py` قبل بدء الاستخدام الفعلي.

## ملاحظات ختامية

- كل الاستدلال (محادثة + تقييم توجيه + تضمين RAG) يمر عبر Ollama
  المحلي فقط — لا مفاتيح API خارجية مطلوبة في أي مكان بالمشروع.
- الأداة الوحيدة التي تتصل فعلياً بالإنترنت هي `get_saudi_stock_price`
  (بيانات سوقية عامة عبر Yahoo Finance)؛ بقية الأدوات في
  `sandbox/tools.py` إما محاكاة (Mock) أو تعمل بالكامل محلياً.
- هذا النظام يقدّم معلومات تعليمية عامة فقط وليس استشارة مالية مرخصة
  — راجع `COMPLIANCE_RULES` في `agents/prompts.py`.
